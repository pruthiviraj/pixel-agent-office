## HOW IT WORKS

One epic in, gated work out. Every box below is a real headless agent with its own prompt, its own log, and its own desk in the office.

1

EPIC

A plain-English goal in a markdown file. That's the whole input.

2

PM PLANS

The PM agent decomposes it into a dependency-ordered task graph.

3

DEVS BUILD

Developer agents implement tasks in parallel, capped by concurrency.

4

QA GATES

A QA agent runs your checks and votes PASS / FAIL. Failures loop back with feedback.

5

LESSONS LEARNED

Each failure is distilled into a one-line lesson injected into future prompts.

## FEATURES

A real orchestration layer with an operator console — not a screensaver.

CONTROL PLANE

Pause, resume, or cancel the sprint from the browser. Retry a failed task, or force-pass / force-fail anything — the engine picks up commands within seconds.

LIVE COST HUD + BUDGET CAPS

Per-worker tokens and dollars, live totals, and a hard cap via ORCH_BUDGET_USD — hit the cap and the sprint winds down cleanly instead of burning quota.

HISTORY & REPLAY

Every run is archived as JSON — board, workers, totals, and a timestamped event log. Browse past sprints and replay how the office got there.

THEMES & PRESENTATION MODE

Swap the office's pixel palette and flip into a clean full-screen presentation view — built for demos, standups, and that wall-mounted TV.

STACK PROFILES

A profile is one small file that briefs the agents on your stack. Ships with software and a worked salesforce example; writing your own takes minutes.

ZERO DEPENDENCIES

Pure Node 18+ stdlib. No npm install, no build step, no framework, no database. Clone it and run it — on Windows, macOS, or Linux.

## QUICKSTART

Node 18+ and the Claude Code CLI on your PATH. Three commands from clone to a live office.

1 · SEE THE OFFICE (FREE — NO AGENTS, NO SPEND)
git clone https://github.com/pruthiviraj/pixel-agent-office.git
cd pixel-agent-office
node server.js   # → open http://localhost:4040/?demo=1
2 · DRY-RUN THE ORCHESTRATOR (MOCKED AGENTS, STILL FREE)
node orchestrate.js --epic ./examples/sample-epic.md --dry-run
3 · RUN A REAL SPRINT ON YOUR PROJECT
node orchestrate.js --project /path/to/your/repo --epic ./my-epic.md
# keep `node server.js` open in another terminal to watch the office

Heads-up: a real sprint spends API quota and edits real files — run it on a branch. Set ORCH_BUDGET_USD=5 for a hard spend cap.

## FAQ

▸How much does it cost to run?
Workers are headless Claude Code agents (claude -p), so a sprint costs whatever those agent sessions cost under your existing Claude Code plan or API key — there is no separate fee and no key stored in this project. The live cost HUD shows tokens and dollars per worker as they run, and ORCH_BUDGET_USD enforces a hard cap: when spend reaches it, no new tasks dispatch and the sprint finishes cleanly. The ?demo=1 office and --dry-run mode are completely free.
▸Is it safe to point at my repo?
Treat it like a fast junior team: real edits, gated by QA. Workers modify files and can run commands in the project you pass with --project, and every change must get a PASS vote from the QA agent before its task counts as done. Best practice: run sprints on a branch, review the diff like any PR, and only enable ORCH_SKIP_PERMISSIONS=1 (full autonomy, no permission prompts) on a repo you trust it to change. You can pause, cancel, or force-fail anything live from the control plane.
▸Which models does it use?
Whatever your claude CLI is configured to use — the orchestrator just spawns headless Claude Code agents and inherits your CLI's model selection and authentication. Switch models in Claude Code and the whole office switches with it. The UI shows each worker's model in the HUD.
▸Do I need npm install or a build step?
No. The entire thing is zero-dependency Node 18+ stdlib — engine, web server, and pixel office included. Clone, node server.js, done. Works on Windows, macOS, and Linux (on Windows, Git for Windows is recommended so workers can spawn).
▸Can it work with my stack?
Yes — the orchestrator is stack-agnostic and a small profile file supplies the stack-specific briefing (how to test, conventions, gotchas). A generic software profile ships by default, with a salesforce profile as a worked example; copying and editing one for your stack takes a couple of minutes.
